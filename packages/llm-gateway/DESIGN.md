# @ludiars/llm-gateway — 汎用トークン節約設計

LUDIARS 全サービス横断の「LLM トークン節約」設計の正本。
本ライブラリ (`@ludiars/llm-gateway`) は、その中の **request 整形レイヤー** を担う transport 非依存の純ロジックである。

- 一般向け解説記事: AIノート「トークン節約」シリーズ (基礎 / 上級 / 運用 / 型)。
- serving 側の詳細: `Famulus/spec/famulus.md`。

---

## 1. 目的とスコープ

LLM 呼び出しのトークン消費を、用途 (コーディング / 議論 / 分類 など) を問わず体系的に削る。
本ドキュメントは **汎用の設計パターン** と、それを LUDIARS の 3 レイヤー (request / serving / consumer) にどう割り付けるかを定義する。

非対象: 個別サービスのプロンプト内容のチューニング、モデル選定そのもの。

---

## 2. トークン燃焼モデル

```
コスト = 入力トークン × 入力単価 + 出力トークン × 出力単価
```

エージェント型ループでは **毎ターン、システムプロンプト + 会話履歴が丸ごと再送される**。
N ターンの会話では、固定部分が N 回課金される。よって削減の主戦場は **出力ではなく「入力の再送」**。

打ち手は 3 レバーに集約され、全ての土台に **計測** が要る。

---

## 3. 汎用設計パターン (3 レバー + 計測)

### ① 再利用する (キャッシュ)

KV キャッシュは **因果律により「先頭からの連続プレフィックスが完全一致」したときだけ** 再利用できる。

| パターン | 内容 | 注意 |
|---|---|---|
| プレフィックス凍結 | 不変 (人格・規約・ツール定義) を前、可変 (履歴末尾・日時・乱数) を後ろ | 冒頭に動的値を 1 個でも置くと以降全滅 |
| 事前ウォームアップ | `max_tokens=0` の空撃ちでキャッシュだけ温める | 連続トラフィックなら不要 |
| フォーク時のプレフィックス流用 | 要約・サブ処理の別呼び出しで親の system/tools を一字一句コピー | 差があると親キャッシュを全外し |

- API キャッシュ (Anthropic 等): cache write ≈ 1.25–2×、cache read ≈ 0.1×。**2 回以上ヒットして黒字**。最小キャッシュ長 (1024–4096 tok) を下回ると無言で非キャッシュ。
- serving キャッシュ (vLLM/SGLang): ブロック単位ハッシュで共通プレフィックスの KV を **全セッションで共有**。並列数に比例して計算・GPU メモリが浮く。

### ② 減らす (選別・圧縮)

| パターン | 内容 | 注意 |
|---|---|---|
| 取得して渡す (RAG/検索) | 全部渡さず関連箇所だけ。コード=リポマップ+検索、資料=意味検索 | 索引の鮮度管理コスト |
| 履歴のローリング要約 | 古いやり取りを要約 1 個に畳む | 要約 LLM 呼び出しのコスト |
| プロンプト圧縮 | 読ませるだけの長文を情報量の薄いトークン間引きで 2–20× (LLMLingua 系) | 会話本体に当てるとニュアンスが死ぬ。**添付資料に限る** |
| context 編集 | 古い/上書き済みツール結果を捨てる | — |
| 出力削減 | 全文再生成でなく diff / 構造化出力で差分だけ | 出力は入力より高単価ゆえ効果大 |

### ③ 逃がす (分散)

| パターン | 内容 | 注意 |
|---|---|---|
| tier ルーティング | 易タスク=安いモデル、難タスクのみ高いモデル | 「念のため全部高い」は規模が掛け算で罰する |
| サブエージェント隔離 | 大量に読ませて結論だけ持ち帰る (生データは親に残らない)。モデル切替の逃げ道にも | — |
| 中間結果を戻さない (PTC) | ツール連鎖をスクリプトに畳み、中間データは戻さず最終出力だけ返す | — |
| Batch | 即答不要処理を非同期バッチへ (≈ 半額) | — |

### 土台: 計測

型を入れる前に計器を付ける。測れないコストは静かに溶ける。

- `cache_read` 率 — プレフィックス共有が効いているか
- トークン / タスク — じわ増えは設計の腐敗
- ルーティング分布 — 高 tier への偏り

## 4. レイヤー責務分担

トークン節約の機構は性質で 3 層に分かれ、置き場所が違う。

```
[consumer] サービス (Discutere / 各 Web フロント / …)
     │  request 整形を LLMClient 境界で適用
     ▼
[@ludiars/llm-gateway]  ← 本ライブラリ。transport 非依存の request 整形
     ├─ remote → Anthropic API 等 (cache_control / tier model)
     └─ local  → [serving] Famulus (vLLM prefix-cache ON で serve)
```

| 層 | 担当 | 実体 |
|---|---|---|
| **request 整形** | プレフィックス順序付け / 履歴要約 / tier 判定 / トークン概算 | **本ライブラリ** |
| **serving** | prefix caching の実有効化 / KV 共有 / routing 実行 / 履歴畳み込み | Famulus (`pickModel`, compaction, system-first 順序, vLLM `--enable-prefix-caching`) |
| **consumer** | 上 2 層の呼び出し・採用 | 各サービス (例: Discutere facilitator の tier 結線) |

**重複させない**: gateway は「送る前の整形ロジック」のみ。HTTP/SDK は呼ばない。serving 側 (Famulus) は同等機能を内製済みのため本 lib に依存しない (ゼロ依存維持)。

## 5. 本ライブラリが提供するもの

| export | 対応パターン | 用途 |
|---|---|---|
| `orderSegments(segments)` | ① プレフィックス凍結 | 断片を安定度順 (fixed→semi→volatile) に整列し、cache breakpoint を算出 |
| `rollingSummary(turns, summarize, opts)` | ② 履歴のローリング要約 | 予算超過時に直近 N を残し古い履歴を要約 (summarizer 注入式) |
| `pickTier(input, rules?)` | ③ tier ルーティング | タスク種別・入力サイズから cheap/strong を判定 |
| `estimateTokens(text)` | 計測の補助 | CJK 重み付けトークン概算 (予算判定用) |

スコープ外 (consumer/serving 側が担う): 実 HTTP 呼び出し、`cache_control` の付与、モデル ID 解決、RAG 索引、プロンプト圧縮 (LLMLingua 連携) は follow-up。

## 6. 採用ガイド

### 迷ったらこの順 (投資対効果順)

1. **計器を付ける** — 現状が見えないと何も始まらない
2. **プレフィックス凍結 (`orderSegments`)** — 修正は小さく効果は全ターンに乗る。最強コスパ
3. **取得して渡す / 履歴要約 (`rollingSummary`)** — 最大の燃焼源を直撃
4. **tier ルーティング (`pickTier`)** — 量で回すほど桁で効く
5. 余力で 圧縮・PTC・Batch・ウォームアップ

### サービス別

- **Claude Code を使う開発**: 検索オンデマンド・diff 編集・キャッシュ・context 編集を既に取得済。新規実装は基本不要。
- **ローカル推論 (Famulus + Gemma 等)**: serving 側で prefix caching を有効化 + `pickModel` で routing。巨大リポのコーディングならリポマップ (②) を前段に検討。
- **自前サービスの推論 (Discutere 等)**: `LLMClient` 境界で本 lib を通す。Discutere は facilitator の重タスクを `pickTier` で strong へ振り分け済 (PR #120)。

---

## 7. 非目標

- プロバイダ固有 API のラップ (本 lib は transport 非依存を保つ)
- プロンプト内容の良し悪し判定
- LLMLingua 等の重い圧縮モデルの同梱 (将来、インターフェース注入で対応)
